# Module 9 — ML Layer: Champion-Challenger (Track 3 add-on)

The submission deck's architecture answer promised a champion-challenger
ML layer — a rule-based scorecard as the score of record (**champion**),
with an Isolation Forest anomaly detector and a Gradient Boosting model
running in parallel (**challengers**), monitored for divergence, never
silently substituted in. This module is that layer, actually built and
run, not just diagrammed.

**Read this first, same as every other module in this pipeline:** two
things below are genuinely useful findings; one is a limitation worth
knowing before you trust a number out of this module in front of a judge
or a real lender.

## Run it

```bash
cd module9_ml_layer
pip install --user scikit-learn   # only new dependency in this whole pipeline
python run_module9.py
```

Produces `ml_output/`:

| File | Description |
|---|---|
| `anomaly_scores.csv` | Isolation Forest `anomaly_score` (0-100, higher = more unusual) + `is_anomaly` per borrower |
| `challenger_scores.csv` | Gradient Boosting `challenger_score` (0-100) per borrower, final model refit on the full population |
| `challenger_holdout_eval.json` | Held-out MAE/R² + permutation feature importance, full 22-feature set |
| `challenger_holdout_eval_robustness.json` | Same, with the one near-deterministic proxy feature excluded — see "Finding 1" below |
| `champion_challenger.csv` | Champion vs. challenger score per borrower, divergence, `flagged_for_review` |
| `validation_summary.json` | Challenger ordering check + anomaly-rate breakdown against the hidden `true_archetype` |

## The one honesty caveat that matters most

**There is no real repayment or default outcome anywhere in this
dataset.** Module 1 generates a *snapshot* of 400 synthetic borrowers, not
a loan-performance history — nobody in this dataset has actually repaid
or defaulted on anything over time. The deck's promise of a challenger
"retrained via ... real repayment outcomes" describes a *production*
version of this module. For this prototype, the hidden `true_archetype`
label (healthy / stagnant / distressed — generated by Module 1 to drive
internal data consistency, and otherwise used only to backtest the rule-
based scorecard's ordering in `module3`/`module5`'s `validate.py`) is
repurposed as the closest available **proxy** for a real outcome label,
via `config.TARGET_BY_ARCHETYPE` (illustrative anchors: healthy=85,
stagnant=55, distressed=20 — not calibrated probabilities of default).
**Swap that one config dict for real observed repayment outcomes the day
real loan-performance data exists; nothing else in this module needs to
change.**

## Finding 1: the full-feature challenger's held-out fit is implausibly
perfect, and that's informative in itself

Running the Gradient Boosting challenger on all 22 features against the
proxy target hits:

```
train_mae: 0.0   train_r2: 1.0
test_mae:  0.0   test_r2:  1.0
```

That is **not** the challenger being unrealistically good — a perfect
held-out fit on real credit data would be a leakage bug, not a success.
Investigated rather than assumed: grouping `borrower_features.csv` by the
hidden `true_archetype` shows `projected_revenue_growth_rate` has **zero
overlap** between archetype groups in this synthetic population —

| Archetype | min | max |
|---|---|---|
| distressed | -0.099 | -0.070 |
| stagnant | 0.009 | 0.015 |
| healthy | 0.096 | 0.153 |

— clean ~8-point gaps on both sides. That's a Module 1 data-generation
artifact (that one feature's archetype-conditioned noise is far tighter
than every other feature's), not a Module 9 bug, and not something a real
noisy dataset would ever hand a model. A sufficiently flexible model can
trivially recover the archetype label from that single feature alone,
which is exactly what permutation importance shows — every other feature
scores 0.0 once that one is available.

**Robustness check** (`challenger_holdout_eval_robustness.json`): refit
with `projected_revenue_growth_rate` excluded —

```
train_mae: 0.76  train_r2: 0.996
test_mae:  1.79  test_r2:  0.974
```

Still very strong, and still dominated by one feature
(`cheque_bounce_rate`, importance 23.9 vs. everything else under 0.2) —
because Module 1's archetype-conditioned generation gives *several*
individual features tight, near-separable per-archetype distributions,
not just the one. **The honest takeaway: high held-out fit in this module
reflects how this synthetic dataset was generated, not evidence the
challenger would perform this well against live, noisy, real-world
features.** The deployed `challenger_score` (see below) still uses the
full feature set deliberately — there's no principled reason to cripple a
production model by removing a legitimately informative feature; the
robustness run exists purely to show genuine multi-feature signal is also
present, not to change what gets deployed.

## Finding 2: the challenger and champion disagree most on distressed borrowers, and the gap is directional

```
                healthy   stagnant   distressed
challenger        85.00      54.93        20.10
champion           74.29      60.10        42.49
```

32.8% of borrowers (131/400) are flagged for review (`|divergence| >=
15`), and the disagreement isn't random noise around zero — it has a
consistent shape: the challenger scores distressed-profile borrowers
**substantially lower** than the champion does (-22.4 points on average),
scores healthy borrowers somewhat higher (+10.7), and roughly agrees on
stagnant ones. A plausible, worth-investigating explanation: the
champion's weighted-average-with-renormalization mechanism (Module 4/5)
smooths a few very weak submetrics against several average ones, while a
tree-based model can learn a sharper "if this cluster of features looks
distressed, score low" rule without that averaging effect. Module 5's own
README already flags its grade bands as uncalibrated against any real
portfolio — this finding is consistent with that same caveat, from an
independent angle.

**Caveat on the caveat:** with `TARGET_BY_ARCHETYPE`'s three coarse
illustrative anchors instead of a continuum of real outcomes, some of
this divergence is mechanical (the challenger's target values were picked
to track `GRADE_BANDS`' band centers, not the champion's own empirical
archetype means) rather than purely a per-borrower disagreement signal.
32.8% flagged is very likely too high a rate to be operationally usable
as-is — a real deployment needs the divergence threshold (and ideally the
target itself) calibrated against real outcomes before a credit manager
should be expected to act on every flag. Reported plainly rather than
tuned down to look more production-ready than it is.

## What this module deliberately does NOT do

- **Never overrides the champion.** `composite_score` (Module 5) stays
  the score of record everywhere else in this pipeline (Modules 6/7/8).
  This module only produces a parallel `challenger_score` and a
  `flagged_for_review` signal for monitoring.
- **Surfaced on the dashboard as advisory-only, with a plain-language
  explanation.** Module 6's dashboard shows this module's output as small
  advisory chips near the composite score ("Model divergence — review
  advised" when `flagged_for_review`; "Unusual profile (ML)" when
  `is_anomaly`) plus a collapsed "ML Model Insights" card. The card now
  includes an auto-generated (template-based, not LLM-generated) 2-3 line
  explanation naming which submetrics most explain the disagreement -
  computed by ranking every scored submetric on how much it drags the
  champion's composite down (dimension weight × submetric weight ×
  (100 - score)) - and tags those specific rows in the scorecard table
  with a small "ML" badge so a reviewer knows exactly where to look.
  Champion score, divergence, and the review flag in the card are LIVE:
  they recompute from the current (possibly overridden) composite score
  on every render, so overriding a flagged submetric and resubmitting
  immediately shows whether that closes the gap with the challenger.
  The challenger score and anomaly score stay frozen at the last Module 9
  run - overriding a submetric does not re-run the trained model, only
  the arithmetic comparison against it. Every override's required
  justification is exported as the JSON feedback artifact intended for
  the challenger's future retraining (see Module 6's override mechanism).
  The dashboard builds fine without this module's output (the card then
  says "ML layer not run"). Note the divergence flag threshold was
  recalibrated 15→25 points for this surface: at 15, a third of all
  borrowers were flagged, which dilutes the flag to noise; at 25, ~9% are
  flagged. Still an assumption pending real-outcome calibration.
  Module 7's mock API still serves the champion score only.
- **Not retrained on override feedback.** The deck describes this as a
  future capability once Module 6's override JSON exports and real
  repayment outcomes exist — this module doesn't consume that feedback
  loop yet, it establishes the two models the loop would eventually feed.
